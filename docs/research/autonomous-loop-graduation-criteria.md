# Autonomous Loop Graduation Criteria

The autonomous execution loop is experimental.

It is not part of the stable alpha promise.

The stable alpha path remains `prometheos work`.

## Current boundary

PrometheOS Lite has two separate surfaces:

| Surface | Status | Source modification |
|---|---|---|
| `prometheos work` / Repo Workbench | stable alpha | no |
| harness execution loop | experimental | possible depending on mode/policy |

## Graduation decision

The autonomous loop must not graduate until there is evidence that it improves outcomes safely and reproducibly.

No promotion should happen merely because the code path exists.

## Required prerequisites

Before promotion, all of the following must be true:

### 1. Model validation plan completed

The validation plan in `docs/research/model-layer-positioning.md` must be run:

- manual local endpoint validation
- harness-value comparison
- small internal benchmark slice

### 2. Minimality enforcement wired into real path

Minimality enforcement must be called from the real execution path before patch application.

It must be tested with fixtures showing:

- unrelated edits are rejected
- excessive changes are rejected
- task-scoped edits pass
- violations are visible in artifacts/evidence

### 3. Escalation ladder built and tested

The loop must distinguish:

- read-only review
- assisted patching
- approval-required patching
- autonomous patching

Each level must have explicit gates.

### 4. Plan-locking / scope-locking exists

The loop must preserve and enforce the approved plan/scope.

It must reject drift such as:

- editing unrelated files
- solving a different task
- expanding beyond approved requirements
- replacing broad files when targeted edits are enough

### 5. Checkpoint and rollback are tested

Promotion requires tests proving:

- checkpoint is created before side effects
- rollback is available after side effects
- rollback is invoked on validation failure when configured
- failure to checkpoint blocks side-effect modes

### 6. Evidence and provenance are complete

Every run must record:

- selected model/provider if model-backed
- whether patch was generated by model/provider/user
- policy gate decision
- dry-run result
- validation result
- review result
- risk result
- minimality result
- checkpoint/rollback result

### 7. User-facing docs are explicit

Docs must clearly say what each mode can and cannot do.

No docs may imply "runs while I sleep" until the above criteria are met.

## What this does not claim

This document does not claim:

- autonomous execution is stable
- the harness beats learned scaffolds
- PrometheOS Lite improves Ornith results
- patch application is safe by default
- benchmark results exist

## Current status

Status: experimental.

Graduation: blocked.

Next required work:

1. Complete model validation plan.
2. Add minimality enforcement to real execution path.
3. Add escalation ladder tests.
4. Add plan-locking/scope-locking.
5. Add checkpoint/rollback promotion tests.

# PrometheOS Lite V1.6 — Harness Engine Implementation Plan

**Version:** 1.6.0  
**Status:** Planning  
**Target Completion:** 11 Weeks  
**Organization:** 5 Epics | 35 Issues | 6 Waves  
**PRD Reference:** `docs/prd/prometheos-lite-v1.6-harness-engine.md`

---

## Executive Summary

This plan delivers the **PrometheOS Lite Harness Engine**: a deterministic, verifiable, autonomous coding execution system that transforms coding tasks into real, tested, reviewable code changes. The implementation follows strict engineering standards—every issue ships with production-ready code, comprehensive tests, documentation, zero TODOs, and clear acceptance criteria.

The PRD restructures V1.6 into **35 sequentially-numbered issues** organized into **5 epics** with **6 execution waves**. This plan reflects that structure.

---

## Epic 1 — Foundation: Repo Intelligence, File Control & Patch Safety

**Theme:** The "eyes" and "hands" of the system—understanding repositories, controlling file access, and applying edits safely.

**Objective:** Enable the harness to intelligently analyze repositories, detect environment capabilities, control file access, and apply structured edits safely.

| Issue | Title | Module | Priority | Dependencies |
|-------|-------|--------|----------|--------------|
| 1 | Repo Intelligence Engine | `src/harness/repo_intelligence.rs` | P0 | None |
| 2 | Environment Fingerprinting | `src/harness/environment.rs` | P0 | None |
| 3 | File Control System | `src/harness/file_control.rs` | P0 | Issues 1, 2 |
| 4 | Edit Protocol | `src/harness/edit_protocol.rs` | P0 | Issue 3 |
| 5 | Patch Applier + Transaction Safety | `src/harness/patch_applier.rs` | P0 | Issue 4 |

**Key Deliverables:**
- `RepoContext` with ranked files, symbol extraction, and relationship mapping
- `EnvironmentProfile` auto-detection for Cargo/npm/pip/go projects  
- `FileSet` with editable/read-only/generated/artifact separation
- `EditOperation` enum supporting 6 edit types with validation
- `PatchResult` with dry-run, transaction safety, structured failure reporting

**Acceptance Criteria:**
- Repo intelligence ranks files by relevance for any task
- Environment detection identifies project type and test commands
- File control rejects edits outside allowed paths
- Edit protocol parses and validates all 6 operation types
- Patch applier applies search/replace with dry-run and transaction safety

## Epic 2 — Execution: Harness Loop, Validation, Repair & Reproduction

**Theme:** The central execution engine that turns WorkContext coding tasks into controlled harness runs.

**Objective:** Create the main execution loop with proper limits, validates results, repairs failures, classifies errors, and manages task-local memory.

| Issue | Title | Module | Priority | Dependencies |
|-------|-------|--------|----------|--------------|
| 6 | Harness Execution Loop | `src/harness/execution_loop.rs` | P0 | Epic 1 |
| 7 | Validation Layer | `src/harness/validation.rs` | P0 | Issue 6 |
| 8 | Failure Taxonomy | `src/harness/failure.rs` | P0 | Issues 6, 7 |
| 9 | Reproduction-First Mode | `src/harness/reproduction.rs` | P1 | Issue 7 |
| 10 | Repair Loop | `src/harness/repair_loop.rs` | P0 | Issues 6, 7, 8 |
| 11 | Task-Local Knowledge Cache | `src/harness/task_cache.rs` | P1 | Issue 6 |
| 12 | Acceptance Criteria Compiler | `src/harness/acceptance.rs` | P0 | Issue 6 |

**Key Deliverables:**
- `HarnessExecutionRequest` with limits (steps, time, cost, attempts)
- `HarnessMode` enum (Review, Autonomous, Benchmark)
- `ValidationPlan` with format/lint/test/repro command execution
- `FailureKind` enum with 12 failure categories
- `RepairRequest` with structured failure analysis
- `ReproductionEvidence` with before/after verification
- `TaskLocalCache` for task-scoped discoveries
- `AcceptanceCriterion` with 4 verification methods

**Acceptance Criteria:**
- Execution stops at max steps, time, cost, or attempt limits
- Validation runs format, lint, tests, and repro commands
- All failures classified with FailureKind for targeted repair
- Repair loop uses failure context to generate fixes
- Bug tasks attempt reproduction before patching
- Task cache stores discoveries without polluting global memory
- Acceptance criteria compiled from requirements and tracked

## Epic 3 — Quality: Review, Risk, Selection, Attempts & Verification

**Theme:** The judgment layer—ensuring only safe, minimal, validated patches are selected.

**Objective:** Multi-layer quality assurance with automated review, adversarial testing, confidence scoring, and risk-based approval gates.

| Issue | Title | Module | Priority | Dependencies |
|-------|-------|--------|----------|--------------|
| 13 | Review Layer | `src/harness/review.rs` | P0 | Issues 5, 7, 12 |
| 14 | Semantic Diff Analyzer | `src/harness/semantic_diff.rs` | P0 | Issue 5 |
| 15 | Patch Minimality Enforcement | `src/harness/minimality.rs` | P0 | Issues 1, 5 |
| 16 | Risk-Based Approval Gates | `src/harness/risk.rs` | P0 | Issues 13, 14, 15 |
| 17 | Verification Strength Levels | `src/harness/verification.rs` | P0 | Issues 7, 9 |
| 18 | Adversarial Validation | `src/harness/adversarial_validation.rs` | P1 | Issues 7, 13, 17 |
| 19 | Confidence Calibration | `src/harness/confidence.rs` | P0 | Issues 7, 13, 17, 18 |
| 20 | Selection Engine | `src/harness/selection.rs` | P0 | Issues 13–19 |
| 21 | Scaling Engine / Attempt Pool | `src/harness/scaling.rs` | P1 | Issues 6, 20 |

**Key Deliverables:**
- `ReviewIssue` with 7 issue types and 5 severity levels
- `SemanticDiff` detecting API/auth/db/config changes
- `MinimalityScore` with unrelated change detection
- `RiskAssessment` with approval requirements for critical changes
- `VerificationStrength` enum with 7 levels
- `AdversarialTestPlan` with generated edge-case tests
- `ConfidenceScore` with 6 weighted factors
- `PatchCandidate` with scoring across 6 dimensions
- `AttemptPool` with Single/Balanced/Max scaling modes

**Acceptance Criteria:**
- Review layer finds bugs, security, performance, maintainability issues
- Semantic diff detects API, auth, DB, dependency, config changes
- Minimality score rejects unrelated changes
- High/Critical risk blocks autonomous commit
- Verification strength reported in all results
- Adversarial tests generated for edge cases
- Confidence reflects validation strength and review cleanliness
- Selection prefers smaller safe patches over large risky ones
- Scaling runs 1/3/N isolated attempts based on mode

## Epic 4 — Infrastructure: Sandbox, Permissions, Git, Trajectory & Observability

**Theme:** Isolated, auditable, reversible, measurable, and replayable execution.

**Objective:** Safe execution environments, complete audit trails, reversible changes, and production observability.

| Issue | Title | Module | Priority | Dependencies |
|-------|-------|--------|----------|--------------|
| 22 | Sandbox Runtime | `src/harness/sandbox.rs` | P0 | Issue 7 |
| 23 | Tool Permission Ledger | `src/harness/permissions.rs` | P0 | Issue 22 |
| 24 | Git Checkpoint System | `src/harness/git_checkpoint.rs` | P0 | Issues 5, 6, 23 |
| 25 | Trajectory Recorder | `src/harness/trajectory.rs` | P0 | Issue 6 |
| 26 | Observability Layer / OpenTelemetry | `src/harness/observability.rs` | P1 | Issues 6, 22, 25 |
| 27 | Runtime Tool Extension | `src/harness/runtime_tools.rs` | P1 | Issues 22, 23 |
| 28 | Time-Travel Debugging | `src/harness/time_travel.rs` | P1 | Issues 24, 25 |

**Key Deliverables:**
- `SandboxRuntime` trait with Local, Deno, and Docker implementations
- `ToolPermission` with 6 permission scopes and approval gates
- `GitCheckpoint` with before/after hashes, diff, rollback capability
- `Trajectory` with step-by-step execution records
- OpenTelemetry metrics (8 harness metrics) and spans (14 spans)
- `TaskLocalTool` with scoped creation, sandboxed execution, cleanup
- `ReplaySnapshot` for post-hoc debugging and replay

**Acceptance Criteria:**
- Sandbox enforces timeouts, captures output, restricts working directory
- Permission ledger records all tool actions, denies unauthorized actions
- Git checkpoint created before patch, supports rollback
- Trajectory records model calls, tool calls, validations, decisions
- Metrics emitted with work_context_id, no secrets in traces
- Runtime tools scoped to WorkContext, cleaned up after
- Time-travel can replay decisions without re-running model calls

## Epic 5 — Intelligence: WorkContext Integration, Memory, Models, Benchmarks & Completion

**Theme:** Connecting the harness to PrometheOS's product layer.

**Objective:** WorkContext integration, model strategy, golden paths, regression memory, benchmarks, artifacts, and evidence-based completion.

| Issue | Title | Module | Priority | Dependencies |
|-------|-------|--------|----------|--------------|
| 29 | WorkContext Integration | `src/work/execution_service.rs`, `src/harness/mod.rs` | P0 | Issues 1–28 |
| 30 | Multi-Model Strategy | `src/harness/model_strategy.rs` | P1 | Issue 6 |
| 31 | Golden Path Templates | `src/harness/golden_paths.rs` | P1 | Issues 6, 12 |
| 32 | Regression Memory | `src/harness/regression_memory.rs` | P1 | Issues 7, 10 |
| 33 | Benchmark Anti-Overfitting | `src/harness/benchmark.rs` | P1 | Issues 20, 21 |
| 34 | Artifact Generator | `src/harness/artifacts.rs` | P0 | Issues 6–20 |
| 35 | Evidence-Based Completion Policy | `src/harness/completion.rs` | P0 | Issues 7, 13, 17, 19, 34 |

**Key Deliverables:**
- `HarnessArtifact` with 6 artifact types attached to WorkContext
- `ModelRole` enum with 5 roles and complexity-based selection
- `GoldenPathTemplate` for 8 common coding tasks
- `RegressionMemory` for storing and retrieving failure patterns
- `BenchmarkTask` with hidden tests and anti-overfitting measures
- `CompletionArtifact` with summary, diff, tests, risk, rollback info
- `CompletionEvidence` with 8 evidence dimensions
- `CompletionDecision` enum (Complete, Blocked, NeedsRepair, NeedsApproval)

**Acceptance Criteria:**
- Coding WorkContext routes to Harness Engine
- Model selection uses cheap models for drafts, strong models for review
- Golden paths guide common tasks with predefined validation gates
- Past regressions influence validation and review
- Benchmarks support private tests and synthetic bug generation
- Completion artifact attached with full handoff documentation
- Evidence-based completion policy gates final status
- No fake completion possible through harness path

## Definition of Done for V1.6

V1.6 is complete only when:

```text
✓ A coding WorkContext can run through the Harness Engine
✓ Repo intelligence identifies relevant files and symbols
✓ Environment fingerprinting detects test/build commands
✓ File control blocks unsafe edits
✓ Structured edits are parsed and validated
✓ A real patch is produced
✓ Patch dry-run works
✓ Git checkpoint is created before patch application
✓ Patch is applied safely
✓ Validation runs format/lint/test/repro commands where available
✓ Repair loop handles failed patch/validation
✓ Review layer produces structured issues
✓ Semantic diff and risk gates run
✓ Patch minimality is scored
✓ Attempt pool can run multiple isolated attempts
✓ Selection engine chooses best candidate
✓ Sandbox runtime enforces timeouts and permissions
✓ Tool permission ledger records actions
✓ Trajectory records every major step
✓ OpenTelemetry/local observability emits traces/metrics
✓ Runtime task-local tools are scoped and auditable
✓ Time-travel replay can inspect decisions
✓ WorkContext stores all harness artifacts
✓ Regression memory records relevant failures
✓ Golden path templates guide common task types
✓ Benchmark mode supports private/synthetic/adversarial tasks
✓ Completion artifact is generated
✓ Evidence-based completion policy gates final status
✓ API and CLI both execute the same code path
✓ All 35 issues have tests
✓ No stubs/placeholders/TODOs in production paths
✓ Docs are complete (7 files)
✓ Changelog is updated
✓ Clean clone passes cargo check/test
```

## Strategic Principles

```text
Keep WorkContext as the moat.
Keep Flow as the substrate.
Make Harness the only path that can touch code.
Make Evidence the only path to completion.
Make Git rollback non-optional.
Make validation mandatory.
Make risk visible.
Make user trust the product because it shows proof.
```

# PrometheOS Lite V1.6 - Harness Engine

## Overview

The Harness Engine is the production-grade, WorkContext-native execution system that transforms coding tasks into **real, tested, reviewable code changes**. Unlike vague code generation systems, the Harness Engine produces actual patches with validation, review, and evidence collection.

## Architecture

### Core Components

The Harness Engine consists of 35 production-ready modules organized into 5 epics:

#### Epic 1: Foundation
- **Repo Intelligence Engine** (`src/harness/repo_intelligence.rs`) - Repository analysis and context building
- **Environment Fingerprinting** (`src/harness/environment.rs`) - Runtime environment detection
- **File Control System** (`src/harness/file_control.rs`) - Atomic file operations
- **Edit Protocol** (`src/harness/edit_protocol.rs`) - Structured edit operations
- **Patch Applier** (`src/harness/patch_applier.rs`) - Transaction-safe patch application

#### Epic 2: Execution
- **Harness Execution Loop** (`src/harness/execution_loop.rs`) - Main execution controller
- **Validation Layer** (`src/harness/validation.rs`) - Multi-level validation system
- **Failure Taxonomy** (`src/harness/failure.rs`) - Structured failure handling
- **Reproduction-First Mode** (`src/harness/reproduction.rs`) - Reproducible execution
- **Repair Loop** (`src/harness/repair_loop.rs`) - Automatic failure recovery
- **Task-Local Knowledge Cache** (`src/harness/knowledge_cache.rs`) - Context caching
- **Acceptance Criteria Compiler** (`src/harness/acceptance.rs`) - Requirement compilation

#### Epic 3: Quality
- **Review Layer** (`src/harness/review.rs`) - Code review and analysis
- **Semantic Diff Analyzer** (`src/harness/semantic_diff.rs`) - Intelligent diff analysis
- **Patch Minimality Enforcement** (`src/harness/minimality.rs`) - Minimal change optimization
- **Risk-Based Approval Gates** (`src/harness/risk.rs`) - Risk assessment and gating
- **Verification Strength Levels** (`src/harness/verification.rs`) - Multi-level verification
- **Adversarial Validation** (`src/harness/adversarial_validation.rs`) - Security validation
- **Confidence Calibration** (`src/harness/confidence.rs`) - Confidence scoring
- **Selection Engine** (`src/harness/selection.rs`) - Best solution selection
- **Scaling Engine / Attempt Pool** (`src/harness/scaling.rs`) - Parallel execution

#### Epic 4: Infrastructure
- **Sandbox Runtime** (`src/harness/sandbox.rs`) - Isolated execution environment
- **Tool Permission Ledger** (`src/harness/permissions.rs`) - Permission management
- **Git Checkpoint System** (`src/harness/git_checkpoint.rs`) - Version control integration
- **Trajectory Recorder** (`src/harness/trajectory.rs`) - Execution tracking
- **Observability Layer / OpenTelemetry** (`src/harness/observability.rs`) - Monitoring and tracing
- **Runtime Tool Extension** (`src/harness/runtime_tools.rs`) - Tool integration
- **Time-Travel Debugging** (`src/harness/time_travel.rs`) - Debug and replay

#### Epic 5: Intelligence
- **WorkContext Integration** (`src/work/execution_service.rs`, `src/harness/mod.rs`) - WorkContext native integration
- **Multi-Model Strategy** (`src/harness/model_strategy.rs`) - Model selection and routing
- **Golden Path Templates** (`src/harness/golden_paths.rs`) - Best practice templates
- **Regression Memory** (`src/harness/regression_memory.rs`) - Learning from past executions
- **Benchmark Anti-Overfitting** (`src/harness/benchmark.rs`) - Performance benchmarking
- **Artifact Generator** (`src/harness/artifacts.rs`) - Result artifact generation
- **Evidence-Based Completion Policy** (`src/harness/completion.rs`) - Completion decision logic

## API Endpoints

### Core Harness Endpoints

#### `POST /api/work-contexts/:id/harness/run`
Execute harness on a WorkContext.

**Request:**
```json
{
  "mode": "auto|assisted|dry_run",
  "repo_root": "/path/to/repo",
  "edits": [
    {
      "type": "create_file|search_replace|delete_file",
      "file": "path/to/file",
      "content": "file content",
      "search": "search pattern",
      "replace": "replacement"
    }
  ]
}
```

**Response:**
```json
{
  "status": "success|failed|incomplete",
  "trajectory": {...},
  "artifacts": [...],
  "confidence": 0.95,
  "risk_assessment": {...},
  "completion_decision": {...}
}
```

#### `GET /api/work-contexts/:id/harness/trajectory`
Retrieve execution trajectory for a WorkContext.

**Response:**
```json
{
  "steps": [
    {
      "step_id": "step_001",
      "timestamp": "2024-01-01T00:00:00Z",
      "action": "analyze_repo",
      "result": {...},
      "duration_ms": 150
    }
  ],
  "total_duration_ms": 1500,
  "success": true
}
```

#### `GET /api/work-contexts/:id/harness/artifacts`
Retrieve generated artifacts for a WorkContext.

**Response:**
```json
{
  "patches": [
    {
      "id": "patch_001",
      "files_modified": ["src/main.rs"],
      "applied": true,
      "rollback_available": true
    }
  ],
  "evidence": [
    {
      "type": "test_result",
      "passed": true,
      "details": "All tests passed"
    }
  ],
  "reviews": [...]
}
```

#### `GET /api/work-contexts/:id/harness/confidence`
Retrieve confidence scores and evidence.

**Response:**
```json
{
  "overall_confidence": 0.92,
  "factors": {
    "test_coverage": 0.95,
    "code_quality": 0.88,
    "risk_level": 0.15,
    "review_score": 0.90
  },
  "evidence": [
    "All unit tests passing",
    "Code follows style guidelines",
    "No security vulnerabilities detected"
  ]
}
```

#### `GET /api/work-contexts/:id/harness/replay`
Replay harness execution from trajectory.

**Request:**
```json
{
  "from_step": "step_005",
  "dry_run": false
}
```

#### `GET /api/work-contexts/:id/harness/risk`
Retrieve risk assessment for the execution.

**Response:**
```json
{
  "overall_risk": "medium",
  "risk_score": 0.35,
  "factors": {
    "code_changes": "low",
    "dependency_impact": "medium",
    "test_coverage": "low",
    "security_impact": "low"
  },
  "recommendations": [
    "Add additional test coverage",
    "Review dependency changes"
  ]
}
```

#### `GET /api/work-contexts/:id/harness/completion`
Retrieve completion decision and evidence.

**Response:**
```json
{
  "decision": "complete|incomplete|blocked",
  "confidence": 0.89,
  "evidence": {
    "acceptance_criteria_met": true,
    "tests_passing": true,
    "review_approved": true,
    "risk_acceptable": true
  },
  "blocking_issues": [],
  "next_steps": []
}
```

## CLI Commands

### Harness Command Structure

The CLI provides comprehensive harness operations under `prometheos work harness`:

#### `prometheos work harness run <id> --mode <mode> --repo-root <path>`
Execute harness on a WorkContext.

**Options:**
- `--mode`: `auto`, `assisted`, `dry_run` (default: `auto`)
- `--repo-root`: Repository root path (default: current directory)

**Example:**
```bash
prometheos work harness run wc-001 --mode auto --repo-root ./my-project
```

#### `prometheos work harness replay <id> --step <step_number>`
Replay harness execution from a specific step.

**Options:**
- `--step`: Step number to replay from (optional)

**Example:**
```bash
prometheos work harness replay wc-001 --step 5
```

#### `prometheos work harness benchmark <id> --type <benchmark_type>`
Run benchmarks on a WorkContext.

**Options:**
- `--type`: `performance`, `accuracy`, `quality` (default: `performance`)

**Example:**
```bash
prometheos work harness benchmark wc-001 --type performance
```

#### `prometheos work harness artifact <id> --type <artifact_type>`
Show artifacts for a WorkContext.

**Options:**
- `--type`: `all`, `patches`, `evidence`, `trajectory` (default: `all`)

**Example:**
```bash
prometheos work harness artifact wc-001 --type patches
```

#### `prometheos work harness risk <id> --threshold <threshold>`
Show risk assessment for a WorkContext.

**Options:**
- `--threshold`: `low`, `medium`, `high`, `critical` (default: `medium`)

**Example:**
```bash
prometheos work harness risk wc-001 --threshold medium
```

#### `prometheos work harness completion <id> --detailed`
Show completion status and evidence.

**Options:**
- `--detailed`: Show detailed completion evidence

**Example:**
```bash
prometheos work harness completion wc-001 --detailed
```

## Integration Examples

### WorkContext Integration

The Harness Engine integrates natively with WorkContext:

```rust
use prometheos_lite::harness::HarnessWorkContextService;
use prometheos_lite::work::WorkContextService;

// Create harness service
let work_context_service = Arc::new(WorkContextService::new(db));
let harness_service = HarnessWorkContextService::new(work_context_service);

// Execute harness
let result = harness_service.execute_harness(
    &work_context_id,
    HarnessMode::Auto,
    edits
).await?;

// Access trajectory
let trajectory = result.trajectory;
let artifacts = result.artifacts;
let confidence = result.confidence;
```

### API Integration

```rust
use axum::routing::{get, post};
use prometheos_lite::api::work_contexts::{
    run_harness, get_harness_metadata
};

// Register routes
app.route("/work-contexts/:id/harness/run", post(run_harness))
    .route("/work-contexts/:id/harness/:view", get(get_harness_metadata));
```

# PrometheOS Lite V1.2 - Operation Layer Documentation

## Overview

The Operation Layer (V1.2) introduces WorkContext lifecycle management, providing a structured framework for tracking, managing, and executing work across different domains. This layer integrates flow execution, artifact management, approval policies, and autonomy levels into a cohesive system.

## Architecture

### Core Components

#### WorkContext
The central entity representing a unit of work with:
- **Identity**: Unique ID, user ID, title, goal
- **Domain**: Work domain (Software, Business, Marketing, Personal, Creative, Research, Operations, General)
- **Lifecycle**: Current phase (Intake, Planning, Execution, Review, Finalization) and status (Draft, InProgress, AwaitingApproval, Completed, Blocked)
- **Autonomy**: Autonomy level (Chat, Autonomous, Review) and approval policy (Auto, ManualAll, RequireForSideEffects)
- **Metadata**: Context type, priority, tags, blocked reason
- **Artifacts**: Generated outputs from flow execution

#### WorkContextService
Service layer for WorkContext CRUD operations:
- `create_context`: Create a new WorkContext
- `get_context`: Retrieve a WorkContext by ID
- `list_contexts`: List all contexts for a user
- `update_context`: Update WorkContext fields
- `update_phase`: Transition to a new phase
- `update_status`: Change the current status
- `add_artifact`: Add an artifact to the context
- `set_blocked_reason`: Mark context as blocked

#### WorkExecutionService
Service for executing flows within WorkContext:
- `execute_flow_in_context`: Execute a flow with context-aware guardrails
- Enforces autonomy level checks
- Enforces approval policy checks
- Maps flow outputs to artifacts
- Updates phases and statuses automatically

#### PhaseController
Manages WorkContext lifecycle transitions:
- `next_phase`: Determines the next phase based on current state
- `can_transition`: Validates if a transition is allowed
- `requires_approval`: Checks if approval is required for a phase transition

#### ArtifactMapper
Converts flow outputs to WorkContext artifacts:
- `map_flow_output`: Maps primary and additional outputs to artifacts
- Supports multiple artifact kinds (Code, Document, Asset, etc.)

#### TemplateLoader
Loads and manages domain profiles from YAML templates:
- `load_all_profiles`: Load all domain profiles
- `load_profile`: Load a specific profile by name
- `get_profile_for_domain`: Get profile for a domain ID
- `save_profile`: Save a profile to disk
- `install_defaults`: Install default domain profiles

### Database Schema

#### work_contexts Table
```sql
CREATE TABLE work_contexts (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    title TEXT NOT NULL,
    goal TEXT NOT NULL,
    domain TEXT NOT NULL,
    context_type TEXT NOT NULL,
    current_phase TEXT NOT NULL,

    autonomy_level TEXT NOT NULL,

    priority TEXT NOT NULL,
    tags TEXT,
    blocked_reason TEXT,
    created_at TEXT NOT NULL,
```

#### artifacts Table
```sql
CREATE TABLE artifacts (
    id TEXT PRIMARY KEY,
    work_context_id TEXT NOT NULL,
    kind TEXT NOT NULL,
    name TEXT NOT NULL,
    content TEXT NOT NULL,
    created_by TEXT NOT NULL,
    storage_type TEXT NOT NULL,
    file_path TEXT,
    created_at TEXT NOT NULL,
    FOREIGN KEY (work_context_id) REFERENCES work_contexts(id) ON DELETE CASCADE

```

```sql
CREATE TABLE decisions (
    id TEXT PRIMARY KEY,
    work_context_id TEXT,

    chosen_option TEXT NOT NULL,

    approved INTEGER NOT NULL DEFAULT 0,
```

```sql
)
```

## WorkContext Lifecycle

### Phases

1. **Intake**: Initial context creation and goal definition
2. **Planning**: Planning flows execute to create execution plans
3. **Execution**: Execution flows perform the actual work
4. **Review**: Review and validation of outputs
5. **Finalization**: Final cleanup and completion

### Statuses

- **Draft**: Context is being defined
- **InProgress**: Active work in progress
- **AwaitingApproval**: Waiting for human approval
- **Completed**: Work is finished
- **Blocked**: Context is blocked from continuation

### Valid Transitions

```
Intake -> Planning -> Execution -> Review -> Finalization
         ^            |            |
         |            v            v
         +------------+------------+
```

### Autonomy Levels

- **Chat**: Requires human confirmation for all actions
- **Autonomous**: Can execute without approval
- **Review**: Requires approval after execution

### Approval Policies

- **Auto**: No approval required
- **ManualAll**: Requires approval for all actions
- **RequireForSideEffects**: Requires approval only for side-effecting operations

## Usage Examples

### Creating a Software Development Context

```rust
use prometheos_lite::work::{WorkContextService, types::WorkDomain};
use std::sync::Arc;

let db = Db::new("prometheos.db")?;
let work_context_service = WorkContextService::new(Arc::new(db));

let context = work_context_service.create_context(
    "user-123".to_string(),
    "Build REST API".to_string(),
    WorkDomain::Software,
    "Create a REST API for user management".to_string(),
)?;
```

### Executing a Flow in Context

```rust
use prometheos_lite::work::WorkExecutionService;

let execution_service = WorkExecutionService::new(
    Arc::new(flow_execution_service),
    Arc::new(work_context_service),
);

let artifact = execution_service
    .execute_flow_in_context(&mut context, "planning.flow.yaml")
    .await?;
```

### Managing Phase Transitions

```rust
use prometheos_lite::work::{PhaseController, types::WorkPhase};

// Check next phase
let next = PhaseController::next_phase(&context);

// Transition to next phase
if let Some(phase) = next {
    work_context_service.update_phase(&mut context, phase)?;
}
```

### Using Templates

```rust
use prometheos_lite::work::TemplateLoader;

let loader = TemplateLoader::default()?;
loader.install_defaults()?;

let profile = loader.get_profile_for_domain("software")?;
```

## Guardrails

### Autonomy Level Enforcement

- **Chat mode**: All executions require human confirmation before proceeding
- **Review mode**: Executions proceed but require approval after completion
- **Autonomous mode**: Executions proceed without approval

### Approval Policy Enforcement

- **Auto**: No approval checks
- **ManualAll**: All phase transitions require approval
- **RequireForSideEffects**: Only side-effecting operations require approval

### Blocked Contexts

Contexts can be blocked with a reason, preventing further execution:
```rust
work_context_service.set_blocked_reason(&mut context, "Security violation".to_string())?;
```
